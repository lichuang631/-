# 移动端抢票安全门、回流热路径与提交节奏设计

## 背景

最近两次复盘暴露了四个问题：

1. 手机离开大麦并进入微信后，脚本仍按固定坐标连点，导致在其他应用中误操作。
2. 订单页显示“尝试提交中”时，OpenCV 仍能匹配底层“立即提交”按钮，造成约每 0.1 至 0.2 秒一次的无效重复点击。
3. 复盘视频只写入发生识别的画面，并按固定帧率连续编码，导致视频时长短于真实运行时间，时间线容易误判。
4. “努力刷新”曾被识别为“继续尝试”；识别后执行的 UI 层级上下文查询在最近一次运行中耗时约 3.2 秒，成为回流热路径的主要阻塞。

本次改动聚焦于阻止跨应用误点、降低无效提交，同时保持开抢热路径速度。验证码仍只做检测和人工暂停，不增加任何绕过行为。

## 目标与性能约束

- 检测到前台不是大麦时，暂停全部自动点击；用户切回大麦后自动恢复。
- 前台应用查询失败时按不安全状态处理，不继续盲点。
- 支付应用进入前台时立即结束自动点击并提示手动支付。
- 每次提交后进入等待态，不在加载遮罩存在期间连续点击“立即提交”。
- 首轮购买点击前只做一次必要的前台确认，不给每个物理点击增加独立查询。
- 正常热路径中的前台查询频率上限为每 0.2 秒一次；同一批三连点仍通过一条 shell 命令发送。
- 非大麦状态的检测延迟目标不超过 0.2 秒。由于前台切换与检查无法原子化，允许检查间隔内已经发出的当前点击批次完成，但发现异常后不得再发新批次。
- scrcpy 视频模式下，回流按钮识别与点击路径不得查询 UI 层级文字；从取得视频帧到发出按钮点击之间只执行 OpenCV 匹配和常量时间的状态判断。
- 移除“努力刷新”点击后的 0.35 秒固定等待，下一次检查统一受视频扫描间隔控制，默认仍为 0.1 秒。
- 不通过一次发送多个可能跨页面的盲点来换取速度；每个回流按钮点击后必须至少读取一张新视频帧，再决定下一次动作。
- 录像处理不得在抢票线程中补写大量重复帧或执行高成本编码工作。

## 方案

### 前台安全门

在 `MobileGrabber` 中增加统一的前台状态判断，状态分为：

- `damai`：允许继续。
- `payment`：停止脚本，交给用户支付。
- `other`：清空“继续尝试”缓存坐标并暂停。
- `unknown`：查询异常，按 `other` 处理，避免盲点。

安全门在开始抢票前强制检查一次。进入热路径后使用最近一次结果，并以 0.2 秒为最短查询间隔。所有能够发出点击的路径，包括购买三连点、OpenCV 按钮、缓存“继续尝试”、票档选择、确认提交和兜底弹窗点击，都必须先经过安全门。

进入暂停时只记录一次日志，随后每 0.2 秒检查前台应用，不发送任何点击。检测到大麦恢复后记录一次恢复日志并继续原状态；暂停时间仍计入 `max_run_seconds`，保证任务不会无限等待。

### 回流视觉热路径

scrcpy 视频模式下，“努力刷新”和“继续尝试”使用纯视觉热路径：

1. 对同一张灰度帧分别匹配两个模板，不按固定顺序命中即返回。
2. 分别限制模板的有效按钮区域，并比较两个候选的匹配分数；只有达到阈值且位置有效的最高分候选可以点击。
3. 使用轻量流程阶段区分购买回流与提交回流。购买阶段不缓存“继续尝试”坐标；提交后明确出现回流按钮时，才允许短期缓存。
4. 删除 `_is_continue_try_context()` 在视觉热路径中的 UI 文本查询。非视频降级模式继续保留 UI 状态识别能力。
5. 点击后立即返回主循环，由 `opencv_scan_interval` 控制下一帧检查，不再额外固定等待 0.35 秒。

缓存坐标只用于已经确认的提交回流按钮。每次缓存点击后必须读取新帧确认，按钮消失、页面变化、前台应用变化或达到次数限制时立即清空缓存。这样避免当前单次 shell 调用与固定 sleep 叠加形成的 0.24 至 0.27 秒机械间隔，也避免在页面切换后继续点击旧位置。

默认扫描间隔保持 0.1 秒，不直接提高截图帧率。优化目标是移除阻塞查询和重复等待，而不是无限提高点击频率。

### 提交防抖

“立即提交”采用一次提交、等待结果的状态约束：

1. 首次匹配并点击“立即提交”后将提交标记为未就绪。
2. 等待期间继续检查支付、验证、售罄、“继续尝试”和“努力刷新”，但不再次点击底层“立即提交”。
3. 只有明确点击了“继续尝试”或“努力刷新”后，才重新允许下一次提交。
4. 等待窗口结束仍无明确状态时，返回现有回流循环；下一轮可再次尝试，避免单次点击未生效后永久卡死。

该策略会移除当前每秒多次的无效提交，减少 ADB 调用与日志写入，不降低弹窗消失后的下一次有效提交速度。

### 复盘时间线

保持录像异步写入和丢帧优先策略，不通过复制空白时段帧来强行拉长视频。录像队列项附带真实采集时间，写入帧时增加相对运行时间标记，便于将画面与 `run.log` 对齐。

时间标记和缩放在录像写线程完成，抢票线程只复制帧并入队。队列满时仍直接丢帧，录像不得反压抢票流程。

## 错误处理

- `app_current()` 暂时失败时暂停点击并限频记录日志，恢复后自动继续。
- 暂停期间检测到用户停止或运行超时，按原有结果退出。
- 录像时间标记失败时继续写原始帧；录像异常不能影响抢票主流程。
- OpenCV 模板缺失或匹配失败时维持现有降级逻辑。

## 测试

建立最小 `pytest` 测试结构，使用假设备和受控时间覆盖：

- 大麦前台允许发送点击批次，且三连点仍只调用一次 shell。
- 0.2 秒节流窗口内复用前台结果，不重复调用 `app_current()`。
- 切到其他应用后不再发送点击，清空缓存坐标；切回大麦后自动恢复。
- 前台查询异常时不点击，支付应用前台时返回支付状态。
- 点击一次“立即提交”后，等待态屏蔽重复提交。
- 点击“继续尝试”或“努力刷新”后重新允许提交。
- 视频热路径匹配两个回流模板后选择有效区域内分数最高的候选。
- 视频热路径不调用 `_is_order_page()`、`_exists_any_contains()` 等 UI 文字查询。
- 回流按钮点击后不执行 0.35 秒固定等待，并且下一次动作必须基于新帧。
- 购买阶段不启用“继续尝试”缓存，提交回流阶段才可启用，页面变化后立即失效。
- 暂停期间仍遵守停止和总超时。
- 录像帧携带真实相对时间，写入异常不会传播到主流程。

测试不连接真实手机，不依赖实际大麦页面。完成后运行全部单元测试，并对关键方法做静态编译检查；若本机仍连接测试手机，再额外测量一次 `app_current()` 延迟，但不在自动测试中操作真实订单。

## 不在本次范围

- 自动拉起或强制切回大麦 APP。
- 验证码自动处理或绕过。
- 票档策略、库存策略和页面模板的大规模重做。
- 对整个移动端流程进行完整状态机重构。
